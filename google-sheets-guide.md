# Google スプレッドシート版 — 記述・運用ガイド

## 全体像

このブックは、要件定義の内容をMarkdownに変換し、AIやエンジニアへ実装の参考資料として渡すために作られている。

タブが分かれているのは、要件を一段階ずつ具体化していくためである。
**アクター** を定め、そこから **ビジネスユースケース（BUC）** を定め、BUCから個々の **ユースケース（UC）** を割り出し、UCの詳細を詰め、そこから **機能要求（FR）** を書き出す。
非機能要求や制約条件は、機能が固まったあとに補足として加える。

1. 📋 概要
2. 📌 前提条件 / 👤 アクター
3. 🎯 ビジネス要求（BR）
4. 📗 BUC
5. 📖 UC
6. ⚙️ 機能要求（FR）
7. 🔒 非機能要求 / 🚧 制約条件 / 🔗 外部IF / ❓ 未解決事項
8. 📝 Markdown 書き出し

ただし、この流れは一方向には進まない。
各タブは互いに参照し合っており、後の工程で気づいた不足は、たいてい前の工程での書き漏らしを意味する。
そのため、後の工程での気づきをもとに前の工程へ戻り、タブ間の整合性を取り直す作業を繰り返す。

この意味で、このブックは記入フォームではなく、前の工程に戻って考え直すことを前提にした思考のツールである。

# 書き方

<a id="guide-tab-roles"></a>

## タブの役割（一覧）

| タブ名 | このシートで書くこと |
|--------|---------------------|
| 📋 概要 | ブック全体の「表紙」。**概要・目的・現状（As-Is）・課題**、スコープ、成功指標（どう評価するか）をここにまとめる。 |
| 📌 前提条件 | この要件が成立するための前提・依存。崩れたときのリスクも記録する。 |
| 👤 アクター | 利用者区分や関係システムなど、要件の文脈に登場する主体の定義。 |
| 🎯 ビジネス要求 | なぜ・何のために作るか、達成したいビジネスの状態を書く。経営・事業側の要求として、優先度とセットで整理する。 |
| 📗 BUC | 事業側の業務単位で整理する。各業務の概要と、紐づくビジネス要求を定義する。 |
| 📙 BUC詳細 | 各業務の手順・行動内容・関連するユースケースを詳述する。|
| 📖 UC一覧 | 誰が・何をしたいかを一覧で追う。|
| 📖 UC詳細 | ユーザー視点のシナリオの本文。前提・基本フロー・代替フロー・例外など、行動の流れを UC ごとに書く。 |
| ⚙️ 機能要求 | システムが具体的に何をするかの振る舞い。画面や API の動きを、要求文と受け入れ基準まで落として書く。 |
| 🔒 非機能要求 | 性能・可用性・セキュリティなど。数値・測定条件まで落とした目標。 |
| 🚧 制約条件 | 使える技術、契約、法規、予算など、設計・実装の前提となる制限。 |
| 🔗 外部IF | 外部システムとの連携。インタフェースの範囲と受け渡しの前提。 |
| ❓ 未解決事項 | まだ決まっていない点。内容、担当、期限を追う。 |
| 📚 用語集 | このプロジェクトで使う用語の定義（読み手の齟齬を防ぐ）。 |
| ✅ 変更履歴 | 仕様書の版と、いつ何が変わったかの記録。 |

<a id="guide-writing-style"></a>

## 記述スタイル

- 目的は「何を作るか」ではなく「何を実現するか」で書きます。
- スコープは「〜すること」の形で動詞を明確にします。
- 成功指標は「それが達成できたか判定できる」形で書きます。測定方法（どうやって測るか）もセットで書きます。
- 機能要求は「[条件] のとき、[主語] は [動作] できる/する」の形で記述し、アクターによって「〜できる」（能力）と「〜する」（動作）を使い分けます。
    - できる → ユーザーの能力（UI で操作できる）
    - する → システムの動作（検証する、保存する）
- 要求は「やること」だけでなく「やらないこと」を書くことで、過剰な実装を防げます。
- 受け入れ基準は「テストケース」の原型です。YesかNoで答えられる形で書きます。
- 非機能要求は、「測定条件」と「測定方法」をセットで書く。
- IDは自動採番です。行を削除しても 欠番を詰めたり番号を再利用したりしません。
    - すでに発行した ID のセル値は **書き換えず**、不要になったときはステータスで「廃止」と記録してください（参照の整合のため）。

# 操作

<a id="guide-id"></a>

## ID の追加と管理

- 新規行は「要求仕様書」→「🧩 行を追加パネル（サイドバー）」から追加する（IDは自動採番される）。
- 手でIDを直した・別ブックからコピーしたあとは、「要求仕様書」→「🔢 IDカウンタをブックから再同期」を実行する。

---

<a id="guide-dropdowns-refresh"></a>
<a id="guide-actor-uc"></a>

## プルダウン参照の更新

- BRやUC、アクターを増やしたり変えたりしたときは、「要求仕様書」→「🔗 入力規則をすべて更新」を実行する。
- BUC の行を手で挿入したときは、E 列（参考：ビジネス要求）に既存行の数式をコピーする（行追加パネル経由なら自動で入る）。
- プルダウンが一覧と合わないときは、タブ名が正しいか確認し、もう一度実行する。

---

<a id="guide-buc-detail"></a>

## BUC 詳細ブロックの追加

1. 📗 BUC で対象行を選び、「要求仕様書」→「📙 選択行の BUC 詳細を追加／更新」を実行する
2. 📙 BUC詳細 に手順表が追加されるので、行動内容を記入する

---

<a id="guide-uc-detail"></a>

## UC 詳細ブロックの追加

1. 📖 UC一覧 で行を選び、「要求仕様書」→「📖 選択行の UC 詳細を追加／更新」を実行する
2. 📖 UC詳細 に項目枠が追加されるので、内容を記入する（見本は UC-001）

---

<a id="guide-markdown"></a>

## Markdown の書き出し

1. スプレッドシートを再読み込みし、メニュー「要求仕様書」→「📝 Markdown を作成（表示・コピー）」を選ぶ
2. ダイアログの「クリップボードにコピー」で取得する
