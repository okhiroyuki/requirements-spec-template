## タブの役割（一覧）

要件はレイヤーが重なります。**ビジネス要求（BR）** は「**なぜ**このプロダクトや施策をやるか／**どんな事業の状態**になれば成功か」、**ビジネスユースケース（BUC）** は事業側の**業務単位**として BR に紐づけて整理し、**ユースケース（UC）** は「**誰が**・**何をしたいか**」という**システム利用のシナリオ**、**機能要求（FR）** は「**システムが何をするか**」という**動作仕様**です。スプレッドシートでは **BR / BUC / UC / FR** のように ID で種類が区別されます。下の表でタブごとの役割を押さえてください。

| タブ名 | このシートで書くこと |
|--------|---------------------|
| 📋 概要 | ブック全体の「表紙」。対象範囲、目的、ステークホルダー、成功指標（どう評価するか）をここにまとめる。 |
| 👤 アクター | 利用者区分や関係システムなど、要件の文脈に登場する主体の定義。 |
| 🎯 ビジネス要求 | なぜ・何のために作るか、達成したいビジネスの状態を書く。経営・事業側の要求として、優先度とセットで整理する。 |
| 📗 ビジネスユースケース | 事業側の業務単位で整理する。各業務の概要と、紐づく **関連BR（🎯 の要求ID）** を書く。 |
| 📖 UC一覧 | 誰が・何をしたいかを一覧で追う。その際ユースケースが、どのビジネス要求に紐づくか・進捗をどう見るかをここで管理する。 |
| 📖 UC詳細 | ユーザー視点のシナリオの本文。前提・基本フロー・代替フロー・例外など、行動の流れを UC ごとに書く。 |
| ⚙️ 機能要求 | システムが具体的に何をするかの振る舞い。画面や API の動きを、要求文と受け入れ基準まで落として書く。 |
| 🔒 非機能要求 | 性能・可用性・セキュリティなど。数値・測定条件まで落とした目標。 |
| 🚧 制約条件 | 使える技術、契約、法規、予算など、設計・実装の前提となる制限。 |
| 🔗 外部IF | 外部システムとの連携。インタフェースの範囲と受け渡しの前提。 |
| 📌 前提条件 | この要件が成立するための前提・依存。崩れたときのリスクも記録する。 |
| ❓ 未解決事項 | まだ決まっていない点。内容、担当、期限を追う。 |
| 📚 用語集 | このプロジェクトで使う用語の定義（読み手の齟齬を防ぐ）。 |
| ✅ 変更履歴 | 仕様書の版と、いつ何が変わったかの記録。 |

## ID の追加と管理

### 操作

- **新規の要求行**: メニュー **「要求仕様書」→「🧩 行を追加パネル（サイドバー）」** を開き、種類（BR / BUC / FR / UC など）ごとに末尾へ 1 行追加すると、通常 **A 列に次の ID** が入ります。
- **手で ID を直した・別ブックからコピーしたあと**は、**「要求仕様書」→「🔢 IDカウンタをブックから再同期」** で、非表示の **`🔢 ID管理`** を実態に合わせてください。

### 自動採番の仕組み

1. **`createRequirementsSheet` を実行**すると、非表示の **`🔢 ID管理`** シートが作成されます。
2. 種類ごと（BR / BUC / FR / UC / NFR / CON / IF / OI / ASM / ACT）に、**ブック内で使われている最大番号**が入ります。サンプル行の ID（例: `BR-001`）から初回も計算されます。
3. **新規行**は行追加パネルで追加するたびに **+1** され、**3 桁ゼロ埋め**（例: `BR-004`、`NFR-012`。1000 以上は桁が増える）で各シートの ID 列に書き込まれます。

## プルダウン参照の更新

**別シートを参照するプルダウンを直したいとき**（BR や UC を増やした、アクター名を変えた、など）は、メニューから **「要求仕様書」→「🔗 入力規則をすべて更新（BR／UC／アクター連携）」** を実行してください。優先度・ステータスなどの固定リストもまとめて張り直されます。行追加パネルで行を足した直後も、該当する入力規則は自動で再走します。

### どの列がどこを見ているか（概要）

| シート | 列 | 参照元 |
|--------|-----|--------|
| 📖 UC一覧 | アクター名 | 👤 アクター の B 列（アクター名） |
| 📖 UC一覧 | 関連BR | 🎯 ビジネス要求 の A 列（`BR-nnn`） |
| 📗 ビジネスユースケース | 関連BR | 🎯 ビジネス要求 の A 列（`BR-nnn`） |
| ⚙️ 機能要求 | 関連UC | 📖 UC一覧 の A 列（`UC-nnn`） |
| 🔗 外部IF | 連携先システム | 👤 アクター の B 列（連携先も 👤 に定義して名前をそろえる） |

これらの列は **リストにある値だけ** 入力できます。まだ選べない名前や ID が必要なときは、先に参照元シートへ行を追加し、上記メニューで入力規則を更新してください。

## UC 詳細ブロックの追加

1. **📖 UC一覧** で行を選び、メニュー **「要求仕様書」→「📖 選択行の UC 詳細を追加／更新」** を選びます。
2. **📖 UC詳細** に、項目枠（事前条件などの見出し・空欄、「基本フロー」「代替フロー」の見出しのみ）が追加されます。フロー表の行番号は自動では入りません。記述の見本は `createRequirementsSheet` が入れる UC-001 を参照してください。

## 要件を書くときの注意（記述スタイル）

- 目的は「何を作るか」ではなく「何を実現するか」で書きます。
    - 開発が進む中で機能が変わっても、目的は変わりません。逆に目的が変わるならそれはその仕様書ではなく新しい内容を作るべきです。
- スコープは「〜すること」の形で動詞を明確にします。
- 成功指標は「それが達成できたか判定できる」形で書きます。
    - 測定方法（どうやって測るか）もセットで書いておくと、のちのテスト計画が圧倒的に楽になります。
- 機能要求は「[条件] のとき、[主語] は [動作] できる/する」の形で記述し、アクターによって「〜できる」（能力）と「〜する」（動作）を使い分けます。
    - できる → ユーザーの能力（UI で操作できる）
    - する → システムの動作（検証する、保存する）
- 要求は「やること」だけでなく「やらないこと」を書くことで、過剰な実装を防げます。
- 受け入れ基準は「テストケース」の原型です。YesかNoで答えられる形で書きます。
- 非機能要求は、「測定条件」と「測定方法」をセットで書く。
- IDは自動採番です。行を削除しても 欠番を詰めたり番号を再利用したりしません。
    - すでに発行した ID のセル値は **書き換えず**、不要になったときはステータスで「廃止」と記録してください（参照の整合のため）。

## Markdown の書き出し

1. スプレッドシートを**再読み込み**し、メニュー **「要求仕様書」→「📝 Markdown を作成（表示・コピー）」** を選びます。
2. ダイアログの **「クリップボードにコピー」**（またはテキストを選択）で取得します。

書き出した Markdown は**生成物**であり、編集の正はスプレッドシートです。

メニュー **「要求仕様書」** が表示されないときは、タブを閉じて開き直すか**再読み込み**してください（`onOpen` でメニューが登録されます）。

## ステークホルダーに共有するときのコツ

ステークホルダーとブックを共有するときの実務上の要点です。

- **共有権限**: **コメント可**（編集は開発側のみ）など、チームで決めます。
- **顧客コメント列**: セルの保護を外し、顧客が書き込めるようにします。
- **変更通知**: スプレッドシートの「通知ルール」で、変更時にメール送信を設定するとよいです。
